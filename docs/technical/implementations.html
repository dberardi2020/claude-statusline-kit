Implementations

The kit's load-bearing abstraction — and the one that isn't code. Two independent programs produce identical output, held together by a shared set of expected-output files rather than by any shared implementation.

Why two, and why no shared core

The obvious fix for dual maintenance is a shared core — one implementation of the logic, two thin platform shims. There is no such core available, because there is no language both shells can call that's guaranteed present on both platforms. Introducing one (Node being the strongest candidate) reintroduces exactly the runtime dependency that ADR 0001 exists to reject: Claude Code's native-binary installer bundles no Node and puts none on PATH, so a Node-based statusline simply fails on machines where Claude Code itself runs fine.

So the duplication is accepted, and the mitigation moves from structure to tests. This is the whole trade:

Two files that must stay in behavioral lockstep, with the lockstep enforced by a CI check rather than by a compiler.

The risk is not hypothetical. Every parity bug found so far has been PowerShell drifting from a correct bash — see the list below.

Bash is the reference

Concretely, that means:

  1. tests/golden/*.txt are generated by running the bash script.
  2. tests/run.ps1 compares PowerShell's output against those same files.
  3. When the two disagree, the default resolution is PowerShell is wrong, unless bash is demonstrably violating the spec.
  4. An intentional format change is made in bash first, the goldens are regenerated from it, and PowerShell is then brought up to the new goldens.

Regenerating (from tests/README.md):

for f in tests/fixtures/*.json; do
  SL_NOW=1700000000 bash statusline-command.sh < "$f" > "tests/golden/$(basename "$f" .json).txt"
done

Per-implementation gotchas

bash — CRLF from a Windows jq

A native Windows jq.exe (Chocolatey's, for instance) terminates lines with CRLF. Under Git Bash every parsed value would then arrive with a trailing \r — enough to break the arithmetic ("5400000\r" - now) and smear stray carriage returns through the output. The script strips CR at both jq boundaries:

… | jq -r '…' | tr -d '\r'                                    # render path
existing_cmd=$(jq -r '…' "$settings" | tr -d '\r')            # install path

Both are no-ops where jq emits LF. The install-path strip matters for a subtler reason: a stray CR would make the kit's own configured command compare unequal to its destination path, so a plain refresh would warn as though it were clobbering a foreign statusline.

This path has no CI coverage (CSK-0004) — it's an unsupported configuration.

bash — the bare-invocation guard

Run with no arguments at an interactive TTY, the script would block forever on cat. It detects that and prints usage instead:

if [ -t 0 ] && [ "$#" -eq 0 ]; then …; fi

PowerShell has no equivalent and falls straight to [Console]::In.ReadToEnd() (CSK-0008).

PowerShell — the UTF-8 BOM is load-bearing

statusline.ps1 carries a UTF-8 BOM on line 1. Without it, Windows PowerShell 5.1 reads the file as Windows-1252, mangles the astral-plane emoji, and fails to parse.

Many editors, formatters, and git filters strip BOMs silently. If the PowerShell script suddenly fails to run on 5.1 and looks unchanged, check the BOM first.

PowerShell — console encoding

The script sets [Console]::OutputEncoding to UTF-8 before writing. The test runner has a matching and much fiddlier problem — PS 5.1 decodes a child's stdout using the OEM codepage, turning every emoji into mojibake — covered in Testing.

PowerShell — the install command is a wrapper

bash installs ~/.claude/statusline-command.sh as the command directly. PowerShell can't: it registers an invocation instead, choosing the executable by the version it's running under —

$exe = if ($PSVersionTable.PSVersion.Major -ge 6) { 'pwsh' } else { 'powershell' }
$newCmd = "$exe -NoProfile -ExecutionPolicy Bypass -File `"$dest`""

-NoProfile keeps a user's profile from injecting output into the statusline; -ExecutionPolicy Bypass avoids a signing prompt on a file the user just downloaded.

Known divergences

Real behavioral differences confirmed by code inspection, none of them covered by the golden suite — because every fixture supplies resets_at and used_percentage together or not at all, and every fixture has a working directory. These are the blind spots the current fixture set structurally cannot see.

The bash column below is verified by execution; the PowerShell column is by inspection only — there's no pwsh on the primary development machine (CSK-0005).

1 · Rate-limit segment drop condition

bashRenders the segment when resets_at is present; appends the percentage only if it is also present (statusline-command.sh:155,159).
PowerShellRequires both resets_at and used_percentage; drops the whole segment otherwise (statusline.ps1:137,148).

Given {"rate_limits":{"five_hour":{"resets_at":…}}} with no percentage:

bash        | 🤖 [Opus 4.8] | ⏳ [2h0m] |
PowerShell  | 🤖 [Opus 4.8] |

Bash matches the spec as written. Whether a countdown without a percentage is desirable is a separate question — but the two implementations should not disagree about it.

2 · Empty-directory placeholder

bash— (U+2014 em dash) when the directory leaf is empty (statusline-command.sh:182).
PowerShell--- (three hyphens) when the working directory is absent (statusline.ps1:166).
bash        | 📁 — | 🌿 --- | 💰 $0.00 | ⏱️ 0m |
PowerShell  | 📁 --- | 🌿 --- | 💰 $0.00 | ⏱️ 0m |

Reachable with a payload carrying neither workspace.current_dir nor cwd. Note the branch segment uses --- in both implementations, so bash is the odd one out — but it's also the reference, which is why this needs a decision rather than a silent fix.

Parity bugs already fixed

All bash-correct, PowerShell drifted — the track record that motivated the golden suite:

  • Rate-limit segments not dropping when the fields were absent.
  • The context bar using banker's rounding where bash rounds half up.
  • The context segment not dropping when its field was absent.
  • cwd not preferring workspace.current_dir over the top-level cwd.

Adding a third implementation

If a platform ever needs one, the checklist is:

  1. Satisfy Rendering exactly — it's the contract, not a description.
  2. Guard every field read; never assert on payload shape.
  3. Honour SL_NOW, or the tests can't pin the clock.
  4. Add a runner that compares against the existing tests/golden/*.txt — do not generate new goldens.
  5. Add a CI job for it, so parity stays merge-blocking.
  6. Support both modes: render and self-install.
